3. Reset reflection answer controllers to original values
4. Remove any newly added (unsaved) answer cards

Update HistoryScreen navigation to EntryDetailScreen to handle the result and refresh data.

**Changes:**

1. Find where HistoryScreen navigates to EntryDetailScreen (the `Navigator.of(context).push(...)` call).

2. Change it to use `await` or `.then()` to capture the result:
```dart
final result = await Navigator.of(context).push<bool>(
  MaterialPageRoute(builder: (_) => EntryDetailScreen(...)),
);
if (result == true) {
  setState(() {
    _historyFuture = _loadData();
  });
}
```

3. The SnackBar for undo is shown by the detail screen using ScaffoldMessenger before popping. The undo action in the SnackBar calls `StorageService.instance.restoreEntry(id)`. After undo completes, history needs to refresh. To handle this:
   - The detail screen should use the ScaffoldMessenger from the history screen's context (passed or captured)
   - Alternative simpler approach: In the detail screen, before popping, capture the ScaffoldMessenger, pop with result, then show the snackbar. The snackbar's undo action should call restoreEntry and then... the history screen won't automatically know about the undo.
   - Best approach: The detail screen shows the snackbar on the history screen's scaffold. When undo is pressed, it restores the entry. The history screen should also rebuild when it detects focus (using RouteAware or simpler: always reload on return regardless of result). Change the condition: always call `_loadData()` when returning from detail, not just when result is true. This handles both edit-save and undo-restore cases.

   Simplified: Always reload on return from detail:
```dart
await Navigator.of(context).push<bool>(
  MaterialPageRoute(builder: (_) => EntryDetailScreen(...)),
);
// Always reload — handles edits, deletes, and undos
setState(() {
  _historyFuture = _loadData();
});
```

Actually, the undo happens asynchronously after pop. Use this pattern instead: make the snackbar undo action call restoreEntry, and since the user will see the entry disappear from history, they tap undo, entry is restored, but history won't update automatically. So add a periodic check or use a callback.

**Simplest robust approach:**
- Always reload when returning from detail (covers edit + delete)
- For undo: pass a VoidCallback to the detail screen that the undo action can call to trigger history refresh
- Add an optional `VoidCallback? onUndoDelete` parameter to EntryDetailScreen
- In history, pass `onUndoDelete: () { setState(() { _historyFuture = _loadData(); }); }`
- In detail screen's undo SnackBar action: call restoreEntry then call widget.onUndoDelete?.call()

This way, when user taps undo on the snackbar (which appears on the history screen's scaffold), history reloads.

**Implementation:**
1. Add `VoidCallback? onUndoDelete` parameter to EntryDetailScreen constructor
2. In history screen, pass the callback when navigating
3. In detail screen, the delete flow:
   - Soft delete entry
   - Capture ScaffoldMessenger reference: `final messenger = ScaffoldMessenger.of(context)`
   - Pop with result true
   - Show snackbar via messenger with undo action that calls restoreEntry + onUndoDelete callback
4. In history screen, always reload on return from detail
